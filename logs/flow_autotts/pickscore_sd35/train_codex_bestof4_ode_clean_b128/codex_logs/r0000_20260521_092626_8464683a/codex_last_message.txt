**Controller**
A beta-smooth scout/confirm/prune/refine policy: low beta stays near deterministic ODE anchoring, mid beta adds preview-guided width plus selective backward children, high beta keeps more contenders alive and branches from the best confirmed anchors.

**Schedule**
- `beta<=0`: deterministic anchor, `latest_active`
- `scouts = 1 + [β>0] + [β>=0.33] + [β>=0.66] + [β>=0.9]`
- `first_probe_time = grid(round((0.18 + 0.10β) * last_idx))`
- `second_probe_time = grid(round((0.50 + 0.18β) * last_idx))`
- `child_time = grid(round((0.82 - 0.22β) * last_idx))`
- `keep_base = 1 + [β>=0.40] + [β>=0.80]`
- `finish_base = 1 + [β>=0.45] + [β>=0.85]`
- `branch_base = [β>=0.35] + [β>=0.75] + [β>=0.95]`
- `noise_policy = inferred_noise` if `β<0.55`, else `mixed_noise`
- `noise_strength = clamp(0.84 - 0.44β, 0.25, 0.85)`

Budget use is monotonic because higher beta increases scout count, keep/finish/branch counts, and shifts refinement earlier enough to make backward/finish chains more expensive.

**What’s new**
This is behaviorally different from the previous best-of-N style controller: it uses preview gaps and uncertainty to decide keep/prune/branch, adds a second confirmation probe, and allocates backward children selectively from ranked anchors instead of finishing every root then refining uniformly.

**Risks**
Preview score calibration may still dominate branch selection; the main expected gains are at beta `0.25`, `0.5`, and `0.75`. Beta `0.0` underuse is justified. Beta `0.25` is intentionally conservative to avoid losing to ODE. Beta `0.5+` was adjusted upward to use more of the target compute.
